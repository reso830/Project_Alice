# Metrics: Hosted Startup Performance

**Feature**: `044-hosted-startup-performance`  
**Scope**: hosted deployment only  
**Created**: 2026-07-07

This artifact records aggregate performance measurements only. It must not include user data, application records, screenshots of private dashboards, or any other PII.

## Measurement Protocol

Field metrics come from Vercel Speed Insights p75 values for the hosted Project Alice deployment: FCP, LCP, CLS, INP, and TTFB.

Lab metrics come from a cold DevTools Performance trace on the hosted deployment with cache disabled, segmented into TTFB, bundle download, parse/execute, `/api/health`, `supabase.auth.getSession()`, and Tracker fetch. `/api/health` must be measured both cold and warm so the serverless cold-start floor is isolated.

Local dev and portable mode are not valid substitutes for this baseline because they do not reproduce the hosted Vercel + Supabase cold-start behavior.

## Residual Risk

Phase 01 has the metrics artifact shape in place, but the baseline is not fully measured yet. Any later "before vs. after" claim must wait until the pending hosted field and lab cells are replaced with direct measurements.

**ACCEPTED (2026-07-07): quantitative FCP/LCP validation deferred to post-deploy production monitoring.** Vercel Speed Insights is a field (RUM) metric by design — it only records real visitor page loads on the **production** domain, not preview deployments. There is therefore no way to obtain a genuine hosted FCP/LCP number before this PR merges and ships to production; every `pending` field/lab cell above reflects that structural limitation, not an oversight. Decision: accept the structurally-verified facts already in this file (loader ships hosted-only, handshake logic covered by C1–C6 tests, bundle-size reduction measured directly, render-blocking font request removed) plus manual/perceived confirmation (branded loader visibly painting before content on a throttled preview load, no regressions found in a local + portable smoke pass) as sufficient for merge. Real FCP/LCP numbers will only be obtainable from production Speed Insights after deploy — follow up post-deploy to fill in the `pending` cells and confirm the expected improvement actually landed.

**RESOLVED (2026-07-07, post-merge): real production Speed Insights data now confirms the improvement.** See the `post-merge (v1.12.0)` row above — FCP 1.87s / LCP 1.98s against the ~8000ms/~13500ms baseline, a real (not estimated) ~4.3x/~6.8x improvement. This is a single early reading (one device/network condition), not yet a stabilized multi-sample p75, so it should be revisited once more production traffic accumulates — but it directly confirms the aggregate effect of WS1–WS5 landed as intended, closing the loop on this residual risk.

**ACCEPTED (2026-07-07): US4 (boot-timeout → Retry) manual browser verification waived.** Attempted on the live PR preview via DevTools network throttling (10 kbps, high latency, 3G presets) — could not force the 10s timeout to fire. Browser throttling approximates real-world conditions but doesn't reliably force a request to hang indefinitely (most throttling/blocking mechanisms fail or complete fast rather than truly stalling); it's structurally a poor tool for reproducing this specific edge case. Decision: rely on `tests/main.test.js`'s fake-timer test instead, which deterministically mocks a health check that never resolves and asserts the Retry affordance appears and reloads correctly — a more reliable verification of this exact path than manual throttling could ever provide.
