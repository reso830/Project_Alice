# Research & Technical Decisions: Hosted Password Management

**Feature**: [plan.md](./plan.md) · **Spec**: [spec.md](./spec.md) · **Date**: 2026-07-10

This records the design decisions behind the plan and the evidence for them. Findings are drawn from `src/data/authStore.js`, `src/services/supabaseClient.js`, `src/pages/welcome/*`, `src/pages/Profile.js`, `src/components/DeleteAccountModal.js`, `server/repositories/supabase/account.js`, and a web search of Supabase's own documented/discussed behavior for recovery-link auth events.

---

## D1 — Recovery-link clicks fire `SIGNED_IN` before `PASSWORD_RECOVERY`; the app must not flash the authenticated shell

**Decision**: Detect a recovery-shaped URL synchronously on load (before any Supabase call); while that guard is armed, suppress routing on a bare `SIGNED_IN` event and wait specifically for `PASSWORD_RECOVERY`, with a short timeout fallback to `recovery-expired`.

**Why**: Supabase's own client emits **two** `onAuthStateChange` events in sequence when a password-recovery link is opened — `SIGNED_IN` first, then `PASSWORD_RECOVERY`. Today's `authStore.init()` callback (`src/data/authStore.js:53-64`) applies every event identically via `applySession(session)`, with no branch on the event name (`_evt` is received but discarded — the leading underscore is the codebase's own "intentionally unused" convention). Unguarded, a recovery-link visit would transiently resolve `authStore` to `status: 'authenticated'` on the `SIGNED_IN` event, and `main.js`'s `render()` (subscribed to every state change) could mount the real app shell for a frame before the second event corrects it — the same class of "flash before the correct destination" bug 044 solved for `local-mode`/ConfigError.

**Evidence**: Web search of Supabase's own GitHub discussions/issues confirms the `SIGNED_IN`-then-`PASSWORD_RECOVERY` event ordering for a recovery-link click. `authStore.js`'s current `onAuthStateChange` callback has no event-name branch. `supabaseClient.js` sets `detectSessionInUrl: true`, so this event pair fires automatically on page load whenever a recovery link is opened — no explicit trigger needed from Alice's code.

**Consequence**: `authStore.init()` reads `location.hash`/`location.search` for Supabase's `type=recovery` marker **before** calling `getSession()` or registering `onAuthStateChange`, arming a guard if present. The guard intercepts a bare `SIGNED_IN` (holds it, does not resolve `authenticated`), resolves `password-recovery` on a confirmed `PASSWORD_RECOVERY`, and resolves `recovery-expired` on a timeout (~5–10s, mirroring 044's boot-timeout pattern) if neither arrives. This single mechanism also implements the spec's expired-link acceptance criterion (AC-7) — a dead/malformed/already-consumed link simply never fires `PASSWORD_RECOVERY`, so the timeout path *is* the expired-link detection, not a separate code path.

**Verification note**: this ordering is drawn from Supabase's own community-reported behavior, not from directly exercising the installed `@supabase/supabase-js@^2.45.0` against a real recovery link in this repo. Plan.md's Validation Approach flags a manual recovery-link test as the first thing to confirm once this phase lands, before the rest of the Reset Password UI is built on top of the assumption.

## D2 — Change Password re-verifies via the anon client, then updates via the admin client (mirrors Delete Account exactly)

**Decision**: `server/repositories/supabase/account.js`'s new `changePassword()` re-verifies `currentPassword` with a fresh anon-key client's `signInWithPassword` (throwing `INVALID_PASSWORD` on failure), then updates the password with the service-role admin client's `admin.auth.admin.updateUserById()`.

**Why**: This is the identical two-step shape `deleteAccount()` already uses and that has already been reviewed/shipped (feature 030) — reusing it outright is lower-risk than inventing a second privilege model for a very similar operation. The account repository is constructed per-request from only `{ userId, email }` (`req.user`), deliberately without a per-request JWT client (`account.js`'s own file comment: "this one does NOT use the caller's per-request JWT client... needs two privileged operations the JWT client cannot perform"). Reusing that same shape for Change Password avoids restructuring how the repository is constructed just to give the update step a lower-privilege path.

**Rejected alternative**: have the server only re-verify (return success/fail), and let the *client* call `supabase.auth.updateUser({password})` directly with its own already-authenticated session (sufficient privilege — a user can always update their own password via their own JWT, no admin client needed for that step). This would use less server-side privilege for the update, but splits Change Password across two round-trips/two code paths (one server verify-only endpoint, one direct client call) instead of the one-request shape Delete Account already established, and the server no longer has a single atomic place to also re-validate the new password's policy compliance server-side. Rejected in favor of consistency with the existing, already-reviewed pattern; noted as a tradeoff in plan.md Risks, not a blocker.

## D3 — Password-policy validation is consolidated into `src/utils/validate.js`

**Decision**: Add `validatePassword(value)` (and the `PASSWORD_MIN = 8` constant) to the existing centralized validation module, and have `LoginForm.js`, `SignupForm.js`, the new `ForgotPasswordForm.js`'s N/A (no password field), and the new `ResetPasswordForm.js`/`PasswordChangeModal.js` all import it, rather than each new form duplicating its own local constant.

**Why**: `src/utils/validate.js` already exists and already centralizes `validateRequired`, `validateEmail`, `validateUrl`, etc. — it is the constitution's named location for "centralized, reusable validation rules." Today's `PASSWORD_MIN = 8` is duplicated verbatim in `LoginForm.js` and `SignupForm.js` (not yet in that module), with two slightly different error strings ("Use at least 8 characters." in the design handoff's copy vs. "Password must be at least 8 characters." in the shipped code) — this feature is a natural point to fix both the duplication and the string inconsistency before adding two more forms that would otherwise duplicate it a third and fourth time.
**Evidence**: `src/pages/welcome/LoginForm.js:7`, `src/pages/welcome/SignupForm.js:6` (both `const PASSWORD_MIN = 8;`); `src/utils/validate.js` has no password-related export today.
**Consequence**: `LoginForm.js`/`SignupForm.js` are touched by this feature as a small refactor (import instead of local const), not a behavior change — same 8-char rule, same UX. Existing tests covering those forms must continue passing unmodified in their assertions.

## D4 — `resetPasswordForEmail`'s `redirectTo` reuses the existing `VITE_AUTH_EMAIL_REDIRECT_URL` (pending confirmation)

**Decision (tentative)**: Pass the same `emailRedirectUrl` export from `supabaseClient.js` (currently used only for `signUp`'s `emailRedirectTo`) as `resetPasswordForEmail`'s `redirectTo`.
**Why**: Supabase requires `redirectTo` to be an allow-listed URL for the project; provisioning a second allow-listed URL (and a second env var) for recovery specifically is avoidable if the existing one is a bare origin/path that Supabase can append its own recovery-specific hash params to, since Supabase's redirect targets are typically just "where to land," with the auth-relevant data arriving as URL fragments/query params Supabase controls, not something baked into the allow-listed URL itself.
**Open item**: this assumes `VITE_AUTH_EMAIL_REDIRECT_URL`'s actual configured value is a generic base URL rather than something that already hardcodes signup-specific routing (e.g., a path baked in that only makes sense post-signup). Needs a direct inspection of the deployed env var value during implementation — not a code change, just a confirm-before-build step (plan.md flags this in Risks and Affected Areas as "inspect only").

## D5 — Explicit sign-out on every exit from Reset Password, not just a successful submit

**Decision**: `ResetPasswordForm`'s success path calls `updateUser({password})` then explicitly `supabase.auth.signOut()` before rerouting, rather than letting the still-valid recovery session carry the user directly into the app shell. **Extended (2026-07-10)**: abandoning the workflow without submitting — close button, Escape, backdrop click, or "Back to sign in" — also explicitly calls `signOut()` before returning to the standard Welcome/login screen, with no success notification.
**Why**: The success-only version directly encodes the spec Clarification (2026-07-10): the recovery session is a valid authenticated session, but the product decision is to end it and require a fresh sign-in with the new password, matching the design handoff's "Back to sign in" affordance. The extension closes a gap a plan review found (flagged by Codex during the plan-review gate, 2026-07-10): the design handoff exposes close/Escape/backdrop/"Back to sign in" on the Reset Password overlay in addition to Submit, and none of the spec/plan/tasks originally said what happens to the active recovery session on those paths. Left unresolved, an abandoned reset could leave `authStore` reporting `password-recovery` (data-model.md §1: this status has no natural expiry of its own — nothing times it out once `PASSWORD_RECOVERY` has been confirmed) while the visible UI had already moved on, the same class of state/UI mismatch the `SIGNED_IN`-before-`PASSWORD_RECOVERY` guard (D1) exists to prevent elsewhere in this feature. Resolved by making **every** exit path — successful or abandoned — end the session the same way, rather than inventing a second, softer invariant for the abandon case.
**Consequence**: `authStore.signOut()` (already exported, already used elsewhere) is reused as-is for both paths — no new sign-out mechanism needed. The post-reset **success** notification uses the existing `setAuthNotice()`/`consumeAuthNotice()` one-shot pattern so it survives the `document.body`-clearing reroute (same mechanism `mountWelcome()` already uses for the delete-account notice); the **abandon** path reroutes the same way but stages no notice (nothing succeeded). Implementation-wise, this means the `reset-password` view's close handling cannot reuse `AuthOverlay`'s generic `close()`/`setView('login')` verbatim (`login`/`signup`/`verification_sent` closing never touches `authStore`) — `reset-password` specifically needs its close/Escape/backdrop/"Back to sign in" handlers routed through a `signOut()`-calling path before/as part of leaving the view (tasks.md T017/T020).
**Rejected alternative**: leave the abandoned recovery session alive until it naturally expires (Supabase's own recovery-session TTL). Simpler (no extra `signOut()` call on the abandon path), but leaves a window where `authStore` and the visible UI disagree about auth state for no product benefit — nothing in the spec wants a "silently still half-signed-in" state to exist, and the whole point of D1's guard machinery is to keep `authStore` and the UI in lockstep during this flow, not just at the happy-path exit.

## D6 — No new server route for Forgot/Reset Password

**Decision**: Both call the Supabase client SDK directly from the browser; no new Express route is added for either.
**Why**: `LoginForm.js`'s `signInWithPassword` and `SignupForm.js`'s `signUp` already establish this precedent — Alice's server is not in the loop for ordinary Supabase Auth operations that don't need elevated privilege or Alice-specific side effects. Change Password is the exception specifically because Alice adds its own current-password re-verification requirement (FR-4) on top of what Supabase requires — Forgot/Reset Password have no equivalent Alice-specific requirement beyond client-side non-enumeration copy (which is a UI concern, not a server one) and the recovery-session validity check (which Supabase itself already enforces — `updateUser` simply fails if the session isn't a valid recovery session).
**Consequence**: Smaller surface area — no new route, no new repository method for these two flows. Keeps the server-side change scoped to exactly the one operation (Change Password) that actually needs it.

## Open Questions (non-blocking; resolve at the owning phase)

- ~~**Abandoned Reset Password sessions**~~ — resolved 2026-07-10, see D5.
- **D1's event-ordering assumption** needs a real recovery-link test against the installed `@supabase/supabase-js` version early in implementation (see plan.md Validation Approach) — the guard/timeout mechanism should be built defensively regardless (it degrades gracefully to `recovery-expired` even if the ordering assumption turns out wrong), but the exact timeout value may need tuning once real timing is observed.
- **D4's redirect URL**: confirm `VITE_AUTH_EMAIL_REDIRECT_URL`'s actual value during WS3 implementation; provision a second env var only if the existing one turns out to be too narrowly scoped.
- **Copy reconciliation**: the design handoff's exact strings ("Use at least 8 characters.", "Passwords don't match.", "Enter a valid email address.") vs. the shipped app's current strings ("Password must be at least 8 characters.") — D3's consolidation is a good forcing function to pick one canonical set; not decided here since it's a copy choice, not an architecture one.
