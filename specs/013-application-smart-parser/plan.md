# Implementation Plan: Smart Application Creation Flow

**Branch**: `013-application-smart-parser` | **Date**: 2026-05-09 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/013-application-smart-parser/spec.md`

## Summary

Intercept the "New Application" button to show a selection overlay (Smart Parser vs Manual Entry). The Smart Parser path accepts pasted job posting text, extracts recognizable fields using client-side heuristics, and pre-populates the existing application form for review. The Manual Entry path preserves the current behavior unchanged. No new API endpoints are required; the parser is a pure client-side utility module.

## Technical Context

**Constraints**: No external network calls during extraction; all parsing runs synchronously client-side
**Scale/Scope**: Single user, local-first; no concurrency concerns

## Architecture

### Data Flow

```
User clicks "New Application"
  → Tracker.onAddApplication()
    → CreationPicker.open({ onApplicationCreate, onApplicationUpdate, onArchiveSuccess })

      [Selection Screen — two cards]
      │
      ├─ Manual Entry chosen
      │    → CreationPicker.close()
      │    → Modal.open(null, { mode: 'create', callbacks })   ← unchanged behavior
      │
      └─ Smart Parser chosen
           → CreationPicker transitions to paste step

           [Paste Step — textarea + Process button]
           │
           ├─ textarea < 20 chars: Process disabled (no action)
           │
           └─ Process clicked
                → loading state shown
                → jobPostParser.parse(text)  ← synchronous, pure
                │
                ├─ ≥1 field extracted:
                │    → CreationPicker.close()
                │    → Modal.open(null, {
                │          mode: 'create',
                │          prefill: parsedData,
                │          callbacks
                │        })
                │
                └─ 0 fields extracted:
                     → error state shown
                     → Retry: stays in paste step, textarea remains editable
                     → Enter manually: CreationPicker.close()
                                       → Modal.open(null, { mode: 'create', callbacks })
```

### Component: CreationPicker

A self-contained overlay module (similar in structure to `Modal.js`) that manages two internal views:

1. **Selection screen** — two option cards rendered side-by-side (desktop) / stacked (mobile). Each card has an icon, title, description, and is fully clickable/keyboard-activatable.
2. **Paste step** — a textarea, a Process button (disabled when textarea < 20 chars), a loading state during extraction, and an error state with retry and manual-entry fallback.

Lifecycle: `CreationPicker.open(callbacks)` / `CreationPicker.close()`. Escape key and backdrop click close the picker. Focus is trapped within the overlay.

### Component: Modal (modified)

`Modal.open()` gains one new optional parameter in its options object:

```
prefill?: Partial<ApplicationRecord>
```

When `mode === 'create'` and `prefill` is provided, the initial draft is:
```
{ ...normalizeApplication({}), status: 'wishlisted', compat: 0, ...prefill }
```

Prefill values are spread last, overriding defaults only where present. All existing call sites pass no `prefill`, so default behavior is fully preserved.

### Module: jobPostParser

A pure function module with no imports from project UI or API layers:

```
parseJobPost(text: string): PartialApplicationRecord
```

Returns an object with whichever fields were successfully extracted. All fields are optional in the return value; unrecognized fields are omitted or set to their normalized defaults. The caller spreads the result over a blank draft.

See `data-model.md` for the complete field-by-field extraction strategy.

### Tracker (modified)

`onAddApplication()` changes from:
```js
Modal.open(null, { mode: 'create', ...callbacks })
```
to:
```js
CreationPicker.open({ ...callbacks })
```

The callbacks (`onApplicationCreate`, `onApplicationUpdate`, `onArchiveSuccess`) are passed through unchanged and forwarded to `Modal.open()` by whichever path the user takes.

## Risks and Tradeoffs

### Risk 1 — Heuristic parser accuracy
**Risk**: Regex and keyword matching will misfire on unusual job post formats. Users may see wrong or empty fields.
**Mitigation**: All pre-filled values are editable before save; existing required-field validation blocks saving incomplete records. Frame the feature as "assistive, not authoritative."
**Residual risk**: Low. User is always in the review loop.

### Risk 2 — Modal.open() prefill coupling
**Risk**: The `prefill` spread happens after `normalizeApplication({})` establishes type-safe defaults, so a prefill value of the wrong type would silently overwrite the safe default.
**Mitigation**: Each extractor in `jobPostParser` is responsible for returning only typed values — string or `''`, positive integer or `null`, valid enum string or `''`, string array. `normalizeApplication` is not called inside the parser; type safety is enforced at the extractor level. The spread therefore always merges correctly-typed values over the normalized blank draft.
**Residual risk**: Very low. Extractor return contracts are enforced by unit tests (T003).

### Risk 3 — CreationPicker and Modal coexistence
**Risk**: If `CreationPicker.open()` is called while a Modal is already open (unlikely but possible from keyboard shortcuts), two overlays could stack.
**Mitigation**: `CreationPicker.open()` checks for an existing open Modal and returns early. This matches the existing `Modal.open()` guard pattern (which calls `close()` before opening).
**Residual risk**: Very low for MVP.

### Risk 4 — CSS scope collision
**Risk**: New `.creation-picker-*` class names could collide with existing styles.
**Mitigation**: Use `creation-picker-` prefix consistently. Audit `main.css` for any existing prefix matches before adding.
**Residual risk**: Negligible — prefix is unique.

### Risk 5 — Salary parsing edge cases
**Risk**: Monthly/annual salary disambiguation is heuristic. Misclassifying monthly as annual (×12) produces a wildly wrong number.
**Mitigation**: Salary field is editable inline in the form. A wrong salary value is immediately visible and correctable. No validation fails on a wrong salary amount.
**Residual risk**: Low — user reviews before saving.
