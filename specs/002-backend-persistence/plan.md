# Implementation Plan: Local Persistence & Backend Support

**Branch**: `002-backend-persistence` | **Date**: 2026-04-26 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `/specs/002-backend-persistence/spec.md`

## Summary

The application currently stores job application records in browser localStorage — data is lost on browser clear or when moving between devices. This plan adds a local SQLite database and an Express API server so that all records persist durably on disk and the frontend reads and writes through the API rather than directly to browser memory.

**Stack**: Node.js + Express (backend) · SQLite via `better-sqlite3` (database) · Zod (validation) · Vite proxy (dev integration) · Vitest (tests)

**Artifact index**: [research.md](research.md) · [data-model.md](data-model.md) · [contracts/api-contract.md](contracts/api-contract.md) · [quickstart.md](quickstart.md)

## Implementation Phases

### Phase 1 — Project Setup

**Goal**: Running Express server, health endpoint, Vite proxy configured, test helpers in place.

Tasks:
- Install runtime dependencies: `express`, `better-sqlite3`, `zod`
- Install dev dependency: `nodemon`
- Add scripts to `package.json`: `server:dev` (nodemon), `server:start` (node), `db:init` (node server/db-init.js)
- Create `server/index.js`: starts Express on port 3001, mounts `/api/applications` router, serves `GET /api/health` returning `{ status: "ok" }`, registers global error handler returning `{ error: { code: "INTERNAL_ERROR", message: "..." } }`
- Create `server/routes/applications.js`: empty Express Router, exported and mounted in `server/index.js`
- Configure Vite proxy in `vite.config.js`: all `/api` requests → `http://localhost:3001`
- Add `data/` and `data/*.db` to `.gitignore`; create `data/.gitkeep`
- Create `tests/server/helpers.js`: exports `makeTestDb()` (temp file via `os.tmpdir()`, runs DDL, returns `{ db, cleanup }`) and `makeMemoryDb()` (`:memory:`) for tests that do not need restart simulation
- Create `server/db-init.js`: imports and calls an explicit `initSchema()` from `server/db.js`; logs `"Database initialized successfully"` on success; exits with code 0 on success, code 1 on error

**Done when**: `npm run server:dev` starts without error; `GET /api/health` returns 200.

### Phase 2 — Foundational (Blocking Prerequisites)

**Goal**: Single source of truth for status values, full 21-column database schema, Zod validation schemas, and repository shell. No user story work can begin until this phase is complete.

Tasks:
- Create `shared/constants.js` at the project root: exports `STATUS_VALUES` array (`wishlisted`, `applied`, `phone_screen`, `interview`, `assessment`, `offer`, `rejected`, `withdrawn`, `ghosted`); update `src/models/application.js` to import from here instead of deriving locally
- Create `server/db.js`: opens `data/alice.db` via `better-sqlite3`; exports `db` instance and explicit `initSchema()` function (schema creation is NOT triggered on module load — it runs only when called)
- Add `CREATE TABLE IF NOT EXISTS applications` DDL inside `initSchema()` with all 21 columns: `id`, `company_name`, `job_title`, `status`, `compat`, `fav`, `source_platform`, `application_date`, `job_posting_url`, `recruiter`, `notes`, `salary`, `responsibilities`, `skills`, `follow_up_action`, `follow_up_date`, `last_status_update`, `created_at`, `updated_at`, `archived`, `metadata` — types and constraints per [data-model.md](data-model.md)
- Add `CREATE INDEX IF NOT EXISTS` statements for `idx_applications_status`, `idx_applications_archived`, `idx_applications_created`
- Create `server/validation/application.js`: imports `STATUS_VALUES` from `shared/constants.js`; defines `toApiError(zodError)` that converts Zod issues to a camelCase-keyed `fields` object with human-readable messages
- Add `createSchema` (Zod): `companyName` (required, min 1), `jobTitle` (required, min 1), `status` (required, enum of `STATUS_VALUES`); `jobPostingUrl` (optional, valid http/https URL); `applicationDate` and `followUpDate` (optional, `YYYY-MM-DD`); `compat` (optional int, 0–100); `fav` (optional boolean); `skills` (optional array of strings); `responsibilities`, `recruiter`, `notes`, `salary`, `sourcePlatform`, `followUpAction` (optional strings); `metadata` (optional, any valid JSON value including null)
- Add `updateSchema`: all fields optional, same rules as `createSchema`; strips `id`, `createdAt`, `updatedAt`, `lastStatusUpdate`, `archived` silently via `.strip()`
- Create `server/db/applications.js`: imports `db` from `server/db.js`; defines `toRecord(row)` (camelCase mapping, `Boolean(row.fav)`, `Boolean(row.archived)`, `JSON.parse(row.skills ?? '[]')`, `row.metadata ? JSON.parse(row.metadata) : null`) and `toRow(fields)` (reverse mapping with `fav ? 1 : 0`, `compat` clamped 0–100, `JSON.stringify(skills)`, `JSON.stringify(metadata)`)

**Done when**: `npm run db:init` completes without error; `data/alice.db` exists with full schema; `shared/constants.js` importable by both frontend and backend.

### Phase 3 — System Property Verification + US1 Base (Priority: P1) 🎯 MVP

**Goal**: Implement the create endpoint and the read endpoints needed to verify the persistence system property. After this phase: POST a record, restart the server, GET the list — the record is still there.

Tasks:
- Write persistence test in `tests/server/persistence.test.js`: uses `makeTestDb()` (temp file); inserts a record; calls `cleanup()` to close the connection; opens a new `better-sqlite3` connection to the same temp file; queries the record; asserts all field values match; deletes the temp file
- Add `create(fields)` to `server/db/applications.js`: maps input via `toRow()`; sets `created_at`, `updated_at`, `last_status_update` to current ISO date (YYYY-MM-DD); runs `INSERT INTO applications`; returns inserted row via `getById(info.lastInsertRowid)`
- Add `getAll()`: `SELECT * FROM applications WHERE archived = 0 ORDER BY created_at DESC`; maps each row with `toRecord()`
- Add `getById(id)`: `SELECT * FROM applications WHERE id = ?`; returns `toRecord(row)` or `null`
- Add `POST /` route to `server/routes/applications.js`: validates body with `createSchema`; returns 400 `VALIDATION_ERROR` with `fields` on failure; calls `create()`; returns 201 `{ data: record }`
- Add `GET /` route: calls `getAll()`; returns 200 `{ data: [...] }`
- Add `GET /:id` route: calls `getById(parseInt(req.params.id, 10))`; returns 200 `{ data: record }` or 404 `{ error: { code: "NOT_FOUND", message: "Application not found" } }`
- Add integration tests in `tests/server/applications.test.js`: `GET /api/health` → 200; `GET /api/applications` on empty DB → `{ data: [] }`; `POST` with valid minimal body → 201 with server-set `id` (integer), `createdAt`, `updatedAt`, `lastStatusUpdate`; `GET /api/applications` after POST → record in `data`; `GET /:id` → record; `GET /9999` → 404

**Done when**: Persistence test passes. `POST` → restart → `GET` returns the record. MVP backend running.

### Phase 4 — US1 Refinement — Full Validation + Frontend Create (Priority: P1)

**Goal**: Full validation error responses on POST, validation unit tests, and the `api.create()` client function.

Tasks:
- Write unit tests in `tests/server/validation.test.js` for `createSchema`: valid full payload passes; missing `companyName`, `jobTitle`, invalid `status` each fail with correct field error; `jobPostingUrl: "ftp://x"` fails; `applicationDate: "2026/04/20"` fails; `compat: 150` is clamped to 100; `skills: "JavaScript"` (not array) fails; `metadata: "string"` (not JSON object/array/null) fails; all optional fields omitted passes
- Add integration tests: `POST` with empty body → 400 `VALIDATION_ERROR` with `fields` containing `companyName`, `jobTitle`, `status`; `POST` with `jobPostingUrl: "not-a-url"` → 400 with `fields.jobPostingUrl`
- Verify `toApiError()` produces camelCase field keys matching the API contract (e.g., `companyName` not `company_name`)
- Create `src/services/api.js`: defines `request(method, path, body)` using `fetch`; catches `TypeError` (network error) and throws `{ code: "NETWORK_ERROR", message: "Cannot connect to the backend — is the server running?" }`; catches non-2xx responses and throws `{ code, message, fields }` from the error envelope; exports named API functions
- Add `api.create(fields)` to `src/services/api.js`: `POST /api/applications`, returns `data` from success response

**Done when**: `POST` with missing `companyName` returns 400 with `error.fields.companyName`. Frontend `api.create()` callable.

### Phase 5 — US4 — Update an Existing Application Record (Priority: P2)

**Goal**: PATCH endpoint with partial update semantics. `lastStatusUpdate` auto-updates only on actual status change. Frontend status changes wired through the API.

Tasks:
- Write unit tests for `updateSchema`: partial payload with only `status` passes; `id` stripped silently; `createdAt` stripped silently; invalid `followUpDate` fails; empty object `{}` passes
- Add integration tests: `PATCH` with `{ status: "interview" }` → 200, updated `status` and `lastStatusUpdate`, all other fields unchanged; `PATCH` with same status → 200, `lastStatusUpdate` NOT changed; `PATCH` with `jobPostingUrl: "bad-url"` → 400; `PATCH /9999` → 404
- Add `update(id, fields)` to `server/db/applications.js`: fetches current record via `getById()`; returns `null` if not found; builds dynamic `UPDATE SET` from supplied keys only, validating each key against a fixed `UPDATABLE_COLUMNS` whitelist (to prevent unexpected column writes); always sets `updated_at` to now; sets `last_status_update` to now only when `status` is present AND differs from the current value; returns updated row via `getById()`
- Add `PATCH /:id` route: validates body with `updateSchema`; returns 400 on failure; calls `update(parseInt(req.params.id, 10), validatedFields)`; returns 404 if `null`; returns 200 `{ data: updatedRecord }`
- Add `api.update(id, fields)` to `src/services/api.js`: `PATCH /api/applications/:id`, returns `data`
- Update `src/pages/Tracker.js`: wire `onStatusChange` to call `api.update(id, { status })` instead of `store.updateStatus()`; show failure toast if API call throws

**Done when**: `PATCH { status: "interview" }` updates `status` + `lastStatusUpdate`; `PATCH` with same status leaves `lastStatusUpdate` unchanged. PATCH on unknown id returns 404.

## Error Handling Summary

| Scenario | HTTP Status | Error Code | `fields` present |
|---|---|---|---|
| Missing or invalid request body fields | 400 | `VALIDATION_ERROR` | Yes |
| Record not found by id | 404 | `NOT_FOUND` | No |
| Unexpected server/DB error | 500 | `INTERNAL_ERROR` | No |

All error responses follow the envelope defined in [contracts/api-contract.md](contracts/api-contract.md). The global Express error handler in `server/index.js` catches unhandled errors and formats them as `INTERNAL_ERROR`.
